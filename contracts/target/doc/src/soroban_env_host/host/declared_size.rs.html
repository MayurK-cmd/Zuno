35// Declared size (bytes) of a single element. This value determines the metering input for clone
36// and comparison. It should be the upperbound (across various compilations and platforms) of the
37// actual type's size. Implementer of the trait needs to decide this value based on Rust's
38// guideline on type layout: https://doc.rust-lang.org/reference/type-layout.html
39pub trait DeclaredSizeForMetering {
40    const DECLARED_SIZE: u64;
41}
42
43macro_rules! impl_declared_size_type {
44    ($t:ty, $size:expr) => {
45        impl DeclaredSizeForMetering for $t {
46            const DECLARED_SIZE: u64 = $size;
47        }
48    };
49}

181// NB: ExtensionPoint is a 1-variant enum with no payload, which Rust optimizes
182// to take zero bytes of memory -- but in XDR it's a 4-byte type like any other
183// union.
184//
185// It exists to help allow the protocol to evolve, as a placeholder type in XDR.
186// ExtensionPoints are sprinkled around the XDR definitions where we expect to
187// need to add stuff in the future. Any field with ExtensionPoint as its type
188// can be _replaced_ in subsequent protocols with some _other_ union type that
189// has a variant with a zero discriminant, and some other non-zero-discriminant
190// variants with new payloads. If old data (written when the field was
191// ExtensionPoint) is read by new code (which knows about the new union type),
192// the new code will just interpret the zero-discriminant old value as the zero
193// case in the new type. All well and good!
194//
195// But none of this works in Rust, because Rust doesn't treat unions (enums) as
196// variable size: it allocates space for the largest variant that can occupy the
197// enum, and it allocates zero bytes for ExtensionPoints. So when you upgrade a
198// union field in XDR, if the new variant is bigger than all existing variants,
199// the Rust size of the field will just change (and since ExtensionPoint is
200// zero-sized, this happens _any_ time you replace an ExtensionPoint with some
201// nonempty type). There's no real getting around this.
202//
203// "Luckily" (from the perspective of deterministic replay) we don't charge the
204// cost model based on Rust's "real sizes" of anything; we charge the cost model
205// based on stable _declared_ sizes we write down explicitly here. So what
206// happens when you upgrade a union field in a way that changes the Rust size of
207// a type is that the Rust size _diverges a bit_ from the declared size: in
208// other words the metering based on declared sizes gets a bit inaccurate, and
209// the test below (`test_declared_size`) that checks real sizes are less than or
210// equal to declared sizes needs to have an exception written into it to handle
211// this divergence.
212//
213// This is not ideal. It means in some cases the metering will be off, at least
214// when reading old data into the new enum (and, say, cloning it) we'll treat it
215// as small when really it is a bit bigger.
216//
217// We can recover a degree of correctness at least in the cases where we're
218// writing new data (occupying the new variants of the extended enum) by
219// essentially pretending that the ext field is a sort of magical smart pointer
220// that occupies zero bytes of its own but points to the new variant as "deep
221// substructure" (the same way we do with real pointer fields).
222//
223// In other words, we can use the following idiom:
224//
225//  - Assume the enclosing struct was `S`, and it previously had an
226//    `ExtensionPoint` `S.ext`, and now it has some other type in `S.ext`. Let's
227//    call the new type we want to add `T` and say we added it by replacing the
228//    `ExtensionPoint` with an enum `E` with variants `V0` (covering the old
229//    `ExtensionPoint`) and `V1(T)`.
230//
231//  - Previously we charged nothing to clone `S.ext`, because it was zero-sized.
232//    So we have to continue to charge nothing when cloning the new field when
233//    it is in its `E::V0` state: that's what the old `ExtensionPoint` data will
234//    be interpreted as when it's deserialized, so we have to treat it as we did
235//    before. This is the case where metering has become incorrect, because `E`
236//    is larger than `ExtensionPoint` but we can't acknowledge that fact.
237//
238//  - To charge nothing for `E::V0`, we actually have to _not_ declare `E` as a
239//    `MeteredClone` / `declared_size` type itself. Rather, inside the body of
240//    `S::charge_for_substructure` we match on `S.ext` and do nothing in the
241//    `E::V0` case, only do nonzero work in the `E::V1` case.
242//
243//  - In the `E::V1` case we can call through to `T::charge_for_substructure`,
244//    treating it almost like it was a separate heap allocation. It still gets
245//    charged for, just "as if" it were out-of-line, rather than inline with `S`
246//    where it happens to be located.
247//
248// This is all a bit weird and awful. It's a mistake that we only noticed after
249// we finalized the metering system of Soroban. We're stuck with it at this
250// point. Hopefully most instances will be fairly benign (and if types get
251// upgraded to their new variants, hopefully also transient!)
252impl_declared_size_type!(ExtensionPoint, 0);

267// RefCell is the underlying data plus an `isize` flag

277impl<K, V> DeclaredSizeForMetering for (K, V)
278where
279    K: DeclaredSizeForMetering,
280    V: DeclaredSizeForMetering,
281{
282    // Their sum plus an 8 bytes of max possible alignment-mismatch overhead. If we need to handle
283    // types with larger alignment mismatches, we need to increase the overhead.
284    const DECLARED_SIZE: u64 = <K as DeclaredSizeForMetering>::DECLARED_SIZE
285        .saturating_add(<V as DeclaredSizeForMetering>::DECLARED_SIZE)
286        .saturating_add(8);
287}

289impl<C: DeclaredSizeForMetering, const N: usize> DeclaredSizeForMetering for [C; N] {
290    // no additional alignment needed since `C::DECLARED_SIZE` is at least `size_of::<C>`
291    // which is already multiple of the alignment
292    const DECLARED_SIZE: u64 = C::DECLARED_SIZE.saturating_mul(N as u64);
293}

551        use more_asserts as ma;
552        use std::mem::size_of;

554        macro_rules! assert_mem_size_le_declared_size {
555            ($t:ty) => {
556                ma::assert_le!(
557                    size_of::<$t>() as u64,
558                    <$t as DeclaredSizeForMetering>::DECLARED_SIZE
559                );
560            };
561            // This variant allows accounting for the case where an
562            // ExtensionPoint has been upgraded to some other type, causing the
563            // enclosing type to exceed its declared size: you can include an
564            // extension type as a second argument and it will be added to the
565            // declared size for the sake of retaining _some_ check here.
566            // Unfortunately this is kinda the best we can do. See the long
567            // comment above around the zero declared_size of ExtensionPoint.
568            ($t:ty, $ext:ty) => {
569                ma::assert_le!(
570                    size_of::<$t>() as u64,
571                    <$t as DeclaredSizeForMetering>::DECLARED_SIZE + (size_of::<$ext>() as u64)
572                );
